# ARMY Morning Brief 제품 요구사항

## 1. 목적

공개 뉴스와 정부·기관 보도자료에서 대한민국 육군 업무에 직접 관련된 항목을 모아 매일
06:30 KST에 짧고 검증 가능한 브리핑을 제공한다. DAPA News와 수집·발행·Telegram 전송
아키텍처는 공유하지만, Army News는 육군·담당 지역 주제와 DAPA와 다른 수집 시간 창을
적용한다. 사용자는 각 항목의 원문 제목, 출처와 HTTPS URL을 통해 즉시 사실을 확인할 수
있어야 한다.

## 2. 대상과 범위

주요 사용자는 출근 전에 육군 관련 정책, 조직·교육훈련, 무기체계·전력화, 작전·안전,
방산·기술 동향을 확인하려는 구성원이다. Army News의 검색 별칭은 `육군`, `8사단`,
`8기동사단`, `3070부대`, `오뚜기부대`이며 담당 지역은 `양주`, `동두천`, `포천`, `연천`,
`의정부`다. 지역 범위에는 음주 사건과 자연재해를 포함하며, 이 두 주제는 지역과 허용
주제가 확인되면 육군 맥락을 요구하지 않는다. 육군 관련 지자체 업무·행사는 관련 군사·육군
맥락이 확인되는 경우에만 포함한다.
정부·국방부·육군·방위사업청 등 공식 공개 자료를 우선하고, 공개 언론 RSS와 Google News
RSS를 보조 출처로 사용한다. 현재 초기 공개 RSS와 우선순위는
`src/army_morning_brief/sources.py`에서 코드로 명시하고 리뷰한다. 환경 설정으로 출처 URL이나
우선순위를 변경하는 기능은 이 범위에 포함하지 않는다.

로그인, 우회 접근 또는 비공개 API가 필요한 출처는 범위에서 제외한다. 모든 원문 URL과
리디렉션 최종 URL은 HTTPS여야 한다.

## 3. 기능 요구사항

### 수집 시간

- 예약 실행은 매일 06:30 KST로 한다.
- 기본 조회 구간은 D-1 14:00 KST부터 D-day 05:00 KST까지로 고정한다.
- 05:00 이후 수집된 기사는 다음 날짜 브리핑으로 넘긴다.
- 06:30 KST에 발행하며, 수동 실행도 이 수집 종료 경계를 적용한다.
- 시간 비교는 KST 시간대 정보가 있는 값으로 수행한다.

### 설정

- 사단의 정식 명칭, 복수 별칭과 담당 지역을 `ARMY_BRIEF_CONFIG_JSON` 설정 데이터로
  관리한다.
- 별칭과 지역 목록은 배포 없이 수정할 수 있어야 한다.
- JSON은 `{"divisions":[{"name":"제8기동사단","aliases":["육군","8사단"],
  "regions":["양주","동두천"]}]}` 형태이며, `BriefConfig.from_mapping`과
  `BriefConfig.from_env`가 비어 있는 값·중복 별칭·잘못된 배열을 거부한다.
- 초기 공개 RSS의 URL·출처 유형·우선순위는 코드 리뷰된 고정 목록이다: 국방부 보도자료
  priority 100, 국방부 공지사항 priority 80, 사단 별칭 Google News RSS priority 60,
  지역 Google News RSS priority 40.
- 사단 별칭·숫자만 일치한 항목과 지역만 일치한 일반 기사는 포함하지 않는다. 지역과
  음주 사건·자연재해가 함께 확인된 기사는 육군 맥락 없이 포함할 수 있으며, 육군 관련
  지자체 업무·행사는 관련 군사·육군 맥락을 추가 확인한다.

### 선별과 출력

- 사단 기사와 육군 관련 지자체 업무·행사는 육군 관련성이 확인되는 카테고리에만
  배치한다. 음주 사건·자연재해 지역 기사는 담당 지역과 허용 주제가 확인되면 배치할 수
  있다.
- 카테고리별 최대 5건으로 제한하되 관련 기사가 부족하면 억지로 채우지 않는다.
- 같은 사건의 재전재는 사건 단위로 묶고 대표 원문 기사 한 건만 제공한다(대표 기사만
  전송).
- 계약 체결/취소, 사업 착수/중단처럼 상태가 다른 사건은 합치지 않는다.
- Telegram HTML 메시지는 아래 세 섹션을 이 순서로 렌더링한다: `🪖 <b>8사단 주요 뉴스</b>`,
  `📍 <b>지역 뉴스</b>`, `🌐 <b>외교·북한 관련</b>`. 각 섹션의 기사 번호는 `1.`부터
  시작하고 다음 섹션에서 다시 `1.`로 시작한다. 빈 섹션은 제목 다음 줄에 `관련 기사 없음`을
  표시한다. 각 항목은 아래 세 논리 줄과 항목 사이 빈 줄을 사용하며, URL 줄은 이스케이프된
  클릭 가능 링크로 렌더링한다.

  ```text
  🪖 <b>8사단 주요 뉴스</b>
  1. 기사 제목 (신문명)
  기사 링크 바로가기
  - 한두문장으로 기사 요약
  ```

  ```text
  📍 <b>지역 뉴스</b>
  1. 기사 제목 (신문명)
  기사 링크 바로가기
  - 한두문장으로 기사 요약
  ```

  ```text
  🌐 <b>외교·북한 관련</b>
  관련 기사 없음
  ```

### 요약 원칙

- 요약은 선택된 단일 원문의 제목·설명·본문에서 명시적으로 확인되는 표현만 발췌한다.
- 여러 기사를 결합해 새로운 사실, 일정, 영향 또는 전망을 생성하지 않는다.
- 불확실한 내용을 확정적으로 바꾸지 않으며 원문에 없는 수치나 관계를 보충하지 않는다.
- 동일 입력과 설정에는 동일한 정렬·중복 제거·요약 결과를 내야 한다.

### 전송과 실행

- 로컬 dry-run은 네트워크 전송 없이 최종 메시지와 원문 URL을 검토 가능하게 한다.
- Telegram Bot Token과 두 수신자 ID가 없어도 dry-run은 동작해야 한다.
- 자격 증명 등록과 실제 Telegram 전송은 dry-run 승인 뒤에만 수행한다.
- `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID`, `TELEGRAM_CHANNEL_ID`는 모두 GitHub Secrets로
  관리하며 실제 값은 절대 커밋하지 않는다. 채팅에 노출된 토큰은 운영 전에 즉시
  폐기·회전한다.
- `TELEGRAM_CHAT_ID`는 기존 개인 대화 수신자, `TELEGRAM_CHANNEL_ID`는 브로드캐스트 채널
  수신자다. 전송 단계는 두 ID를 쉼표로 결합해 동일한 브리핑을 두 수신자에게 순서대로
  보낸다.
- 두 수신자 ID가 존재하기 전에는 실제 전송을 수행하지 않는다.
- 예약 실행과 수동 실행을 구분해 기록하며 같은 날짜의 예약 메시지 중복 전송을 막는다.

## 4. 안전·보안 요구사항

- 공개적으로 접근 가능한 기사와 공식 자료만 처리한다.
- 부대의 세부 위치, 인원, 이동 경로, 작전 계획, 장비 배치를 수집하거나 추론하지 않는다.
- 기사에 없는 병력 규모, 준비태세, 취약점 또는 향후 행동을 생성하지 않는다.
- 운영용 Bot Token과 개인 대화·브로드캐스트 채널 수신자 ID는 GitHub Actions Secrets로만
  주입하며 코드·문서·로그에 실제 값을 남기지 않는다.
- 로그는 자격 증명과 메시지 수신자 식별자를 마스킹한다.

## 5. 비기능 요구사항

- Python 3.12 및 `uv` 기반의 src-layout 패키지로 제공한다.
- HTTP 호출은 명시적 타임아웃, 제한된 재시도와 개별 출처 오류 격리를 적용한다.
- transient HTTP 상태·transport 오류는 최대 2회 재시도하고 출처별 60초 수집 예산과
  재시도 대기 상한을 적용한다.
- 일부 출처 실패가 전체 브리핑 생성을 막지 않아야 한다.
- 단위 테스트, fixture 기반 회귀 테스트, Ruff와 basedpyright 검사를 통과해야 한다.
- 외부 서비스가 없는 테스트와 dry-run으로 핵심 결과를 재현할 수 있어야 한다.

## 6. 완료 조건

- `ARMY_BRIEF_CONFIG_JSON` 예시가 위 스키마로 파싱되고, 설정된 사단 별칭은 육군 업무
  문맥을 검증하며 지역 기사는 허용 주제와 지자체 업무·행사
  별 관련 맥락 규칙을 적용해 오탐하지 않는다.
- D-1 14:00 KST부터 D-day 05:00 KST까지의 경계 테스트가 통과한다.
- HTTP가 아닌 출처와 민감 정보 추론 항목이 거부된다.
- 같은 사건의 중복은 한 건만 남고 상태가 다른 사건은 분리된다.
- canonical URL·사건 anchor를 사용하는 이벤트 중복 제거가 서로 다른 언론사의 재전재를
  대표 원문 한 건으로 줄이고 국가·지역·상태가 다른 사건은 유지한다.
- 각 Telegram 섹션은 정확한 이모지·굵은 제목을 사용하고, 항목은 섹션별 `1.`부터의 번호,
  짧은 원문 링크 바로가기, 발췌 요약의 세 줄과 빈 줄 구분을 지킨다. 빈 섹션은 `관련 기사
  없음`을 표시한다.
- 요약의 모든 문장이 선택된 원문에서 추적 가능하다.
- 자격 증명 없이 dry-run이 성공한 뒤에만 실제 전송 검증을 수행한다.
- GitHub Actions의 `30 21 * * *` UTC 예약, `workflow_dispatch`, 고정 concurrency와
  날짜별 예약 중복 방지 계약 테스트가 통과한다.
- 전체 `pytest`, `ruff check`, `ruff format --check`, `basedpyright` 검사가 통과한다.
